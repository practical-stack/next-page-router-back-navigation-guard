history-augmentation.ts
몽키패칭과 initialize 로직 완전 해설

이 파일은 라이브러리의 가장 밑바닥 기반 계층입니다. 브라우저의 history.pushState / history.replaceState를 몽키패칭(monkey-patching)해서, History API가 원래 제공하지 않는 "내가 history 스택의 몇 번째에 있는가"와 "지금 세션이 어느 세션인가"를 추적합니다. 이 문서는 모듈의 구성 요소(상수·헬퍼·팩토리)를 먼저 본 뒤, initializeHistoryStateSyncOnce()가 그것들을 어떻게 엮어 무엇을, 왜, 그리고 그 결과가 어디서 소비되는지를 단계별로 풀어 설명합니다.

1. 왜 몽키패칭이 필요한가

이 라이브러리의 목적은 뒤로 가기(back)만 가로채는 것입니다. 그런데 브라우저가 주는 popstate 이벤트는 결정적인 정보가 빠져 있습니다.

핵심 난점 — popstate는 "뒤로"인지 "앞으로"인지 알려주지 않는다
사용자가 뒤로 가기를 눌러도, 앞으로 가기를 눌러도 둘 다 똑같이 popstate가 발생합니다. 이벤트 객체에는 방향(delta)이 없습니다. 우리는 뒤로 가기만 막고 싶은데, 방향을 모르면 막을 수가 없습니다.

그럼 위치 인덱스를 어디서 구하나?

방법인덱스 제공?문제
history.length❌스택의 총 길이일 뿐, 현재 위치가 아니다
Navigation API navigation.currentEntry.index✅Safari / Firefox 미지원 → 못 씀
history.state에 내가 직접 인덱스를 심기✅모든 브라우저 동작 → 이 라이브러리가 선택한 길
이유
표준 API로는 "현재 위치"를 알 방법이 전무합니다. 그래서 pushState/replaceState를 가로채서, 호출될 때마다 우리가 매긴 위치 번호를 history.state 안에 몰래 기록합니다. 이것이 몽키패칭을 하는 단 하나의 이유입니다. (파일 상단 주석 history-augmentation.ts:1-14에 동일한 요지가 적혀 있습니다.)

2. 주입하는 메타데이터 2개의 키

패칭된 함수들은 모든 history 진입점(entry)의 history.state에 다음 두 개의 키를 추가합니다. 키 이름은 파일 상단의 상수로 정의돼 있습니다.

19 const SESSION_TOKEN_KEY = "__next_session_token";
20 const STACK_INDEX_KEY = "__next_navigation_stack_index";
키의미왜 필요한가
__next_navigation_stack_index 이 history 진입점이 스택에서 몇 번째인가 (0부터 증가) popstate 시 delta = 다음 인덱스 − 현재 인덱스를 계산해서
음수면 뒤로, 양수면 앞으로를 판별
__next_session_token 이 페이지 로드(세션)의 고유 난수 ID 세션 경계(외부/구 진입점)를 감지. 토큰이 다르면
인덱스를 신뢰할 수 없는 상황(=Session Boundary). 새로고침은 토큰이 복원되어 경계가 아님
한 줄로: 브라우저의 각 history 진입점에는 history.state 객체가 딸려 있습니다. 몽키패칭은 진입점이 만들어질 때마다 그 history.state에 위치 인덱스와 세션 토큰을 기록해 둡니다. 나중에 popstate가 오면 도착한 진입점의 history.state를 읽어 "몇 번에서 몇 번으로 이동했는지"를 계산해 방향을 알아냅니다.

모듈 골격 — 싱글톤 상태 4개

파일 최상단에 모듈-평가 시점에 캡처되는 상수 1개와 모듈 레벨 변수 3개가 있습니다. 이들은 앱 전체에서 단 하나만 존재합니다.

31 // 이전 세션 토큰/인덱스를 모듈-평가 때 캡처 (불변 상수)
31 const _capturedSessionStateAtModuleLoad: RenderedState | null = …;
   
43 // Module-level singleton state
43 let _isHistoryStateSyncInitialized = false;                       // 패칭 1회 가드
44 let _renderedState: RenderedState = { historyIndex: -1, sessionToken: "" };
45 let _setRenderedStateAndSyncToHistory = () => {};            // 나중에 진짜 구현으로 교체
  • _capturedSessionStateAtModuleLoad — 모듈이 평가되는 순간 history.state에서 읽어낸 이전 세션의 { sessionToken, historyIndex } (없으면 null). 한 번 잡으면 바뀌지 않는 상수. 새로고침이 세션을 끊지 않게 만드는 핵심 — 아래 전용 절.
  • _isHistoryStateSyncInitialized — 패칭을 딱 한 번만 하기 위한 잠금장치. 초기값 false, 끝에서 true.
  • _renderedState — 지금 화면에 렌더링된 페이지가 가진 { historyIndex, sessionToken }. 우리가 추적하는 "현재 위치"의 진실(source of truth). 초기값은 의도적으로 무효한 { -1, "" }.
  • _setRenderedStateAndSyncToHistory — 초기엔 빈 함수(no-op). initialize 안에서 실제 구현으로 교체되고, 밖으로 반환되어 popstate 핸들러가 사용합니다(④).

모듈-평가 토큰 캡처 — _capturedSessionStateAtModuleLoad

이전 세션의 토큰을 모듈 평가 시점(module-evaluation time)에 캡처해 둡니다. 이것이 새로고침을 "끊긴 세션"이 아니라 "이어지는 세션"으로 다루게 해주는 장치입니다. 

31 const _capturedSessionStateAtModuleLoad: RenderedState | null =
32   typeof window !== "undefined" &&
33   window.history &&
34   window.history.state &&
35   window.history.state[SESSION_TOKEN_KEY]
36     ? {
37         sessionToken: window.history.state[SESSION_TOKEN_KEY],
38         historyIndex: Number(window.history.state[STACK_INDEX_KEY]) || 0,
39       }
40     : null;
이유
브라우저는 새로고침(reload)을 가로질러 그 진입점의 history.state를 대략 load 이벤트 전까지 보존합니다. Next.js Pages Router는 그보다 약간 늦게(router hydration 시, Provider가 마운트될 때) 그것을 덮어씁니다. 그런데 모듈 평가는 hydration보다 먼저 실행되므로, 그 순간엔 이전 토큰이 아직 history.state에 남아 있습니다 → 캡처 가능합니다. (주석 history-augmentation.ts:22-30에 같은 설명이 있습니다.)
시간축 ───────────────────────────────────────────────► reload 직후: history.state 에 이전 세션 토큰이 아직 남아 있음 │ ├─ ① 모듈 평가 ← 여기서 토큰 캡처 (_capturedSessionStateAtModuleLoad) ✓ │ ├─ ② Next router hydration / Provider 마운트 ← 여기서 history.state 덮어씀 ✗ │ (만약 여기서 읽으면 token-less 한 Next 상태만 보임) ▼ ③ initializeHistoryStateSyncOnce() ← 캡처해 둔 토큰을 복원
효과: 새로고침된 진입점은 원래 세션에 재합류하고, back/forward는 인덱스(delta) 기반으로 정상 구분됩니다. 더 이상 새로고침이 모든 popstate를 세션 경계로 오인하게 만들지 않습니다. (과거의 "forward-after-refresh 오인" 한계는 이 모듈-평가 복원으로 해결되었습니다.)

generateSessionToken — 새 세션에만 새 토큰

57 export function generateSessionToken(): string {
58   return Math.random().toString(36).substring(2);
59 }
매 로드마다 새 토큰이 아니다 — 복원할 게 없을 때만 새로 만든다.
이전 세션 토큰은 _capturedSessionStateAtModuleLoad로 이미 캡처해 둡니다(위 절). 그래서 generateSessionToken()은 복원할 세션이 없을 때만 호출됩니다 — 진짜 첫 방문(③의 else 분기), 또는 토큰이 아직 없는 상태에서의 push/replace(팩토리의 안전망). 새로고침은 mismatch가 아니며, back/forward는 인덱스(delta)로 정상 구분됩니다.

3. 패치를 만드는 작은 헬퍼 3개

패칭 로직 자체는 createPatchedHistoryMethod 팩토리(다음 절)가 담당하지만, 그 전에 알아둘 작은 순수 함수 3개가 있습니다.

64 // 현재 렌더링 상태의 불변 복사본
64 export function getRenderedState(): RenderedState {
65   return { ..._renderedState };
66 }
   
71 // 모듈 상태를 불변 업데이트 (포인터 이동의 유일한 창구)
71 function setRenderedState(renderedState: RenderedState): void {
72   _renderedState = { ...renderedState };
73 }
   
78 // history.state 객체에 우리 추적 메타데이터를 병합
78 function withTrackingMetadata(historyState, { sessionToken, historyIndex }) {
82   return {
83     ...historyState,                          // 호출자/Next가 넣은 state 보존
84     [SESSION_TOKEN_KEY]: sessionToken,
85     [STACK_INDEX_KEY]: historyIndex,
86   };
87 }
헬퍼하는 일핵심 포인트
getRenderedState() 모듈 포인터 _renderedState의 복사본 반환 export되어 rendered-state-context.ts가 그대로 노출 (외부의 읽기 창구)
setRenderedState() 모듈 포인터를 불변 교체 "현재 위치"를 옮기는 단 하나의 통로. 팩토리·sync 클로저·복원 분기가 모두 이걸 호출
withTrackingMetadata() 임의의 state에 {token, index} 두 키를 덧씌움 스프레드로 원본 보존 → Next가 넣은 {url, as, key, __N} 등이 사라지지 않음
이 3개가 "상태 진실"의 두 축을 각각 담당합니다 — setRenderedState는 메모리 포인터(_renderedState)를, withTrackingMetadata는 디스크 메타데이터(history.state)를 다룹니다. 이 구분이 sync ON/OFF 이해의 토대입니다.

4. createPatchedHistoryMethod — push/replace를 한 틀로

예전 구현은 pushState와 replaceState를 각각 따로 인라인으로 패칭했습니다. 지금은 둘의 차이가 "인덱스를 +1 하느냐 마느냐" 단 하나뿐이라는 점에 착안해, advanceIndex 불리언 하나로 갈라지는 단일 팩토리로 합쳤습니다. 

96 function createPatchedHistoryMethod(
97   original: History["pushState"],
98   advanceIndex: boolean          // push → true, replace → false
99 ): History["pushState"] {
100  return function (this: History, historyState, unused, url) {
101    const current = getRenderedState();
102    const next: RenderedState = current.sessionToken
103      ? {                                                          // 정상: 세션 유지
104          sessionToken: current.sessionToken,
105          historyIndex: current.historyIndex + (advanceIndex ? 1 : 0),
106        }
107      : { sessionToken: generateSessionToken(), historyIndex: 0 }; // 안전망
108
109    setRenderedState(next);                                       // ① 포인터 갱신
110    debug(…);
113    original.call(this, withTrackingMetadata(historyState, next), unused, url); // ② 원본 위임
114  };
115}
구현
호출되면 — ① 현재 상태를 읽어 next를 계산(토큰 유지, 인덱스는 advanceIndex면 +1) → ② setRenderedState로 포인터를 옮기고 → ③ withTrackingMetadata로 메타데이터를 끼운 state를 원본에 위임합니다. this를 그대로 넘겨 호출 문맥을 유지합니다.
인자매핑의미
advanceIndex: true→ pushState로 사용새 진입점이 생기므로 인덱스 +1
advanceIndex: false→ replaceState로 사용같은 칸 덮어쓰기이므로 인덱스 유지
왜 안전망(: { generateSessionToken(), 0 })이 있나? 정상 흐름에선 initialize가 먼저 토큰을 채우므로 current.sessionToken은 항상 truthy입니다. 하지만 어떤 이유로든 토큰이 빈 상태에서 push/replace가 먼저 불리면, 추적이 깨지지 않게 그 자리에서 새 세션을 시작합니다. replaceState(advanceIndex=false)에서도 이 분기는 인덱스를 0으로 시작합니다.
누가 pushState/replaceState를 부르나? Next.js의 router.push(), <Link> 클릭, 그리고 Next 내부 라우팅이 호출합니다. 즉 사용자가 라우팅할 때마다 이 패치가 자동으로 끼어들어 메타데이터를 기록합니다. (이 라이브러리가 직접 부르는 게 아닙니다.)

5. initializeHistoryStateSyncOnce 단계별 분해

이 함수가 이 파일의 심장입니다(history-augmentation.ts:126-200). 위에서 본 부품들을 6단계로 엮습니다.

① 싱글톤 가드 — 두 번 패칭하지 않는다

132if (_isHistoryStateSyncInitialized) {
133  return { setRenderedStateAndSyncToHistory: _setRenderedStateAndSyncToHistory };
134}
목적
이미 패칭이 끝났다면, 새로 패칭하지 말고 기존 함수만 돌려준다.
이유
createRenderedStateContext()는 popstate 핸들러가 생성될 때마다 이 함수를 호출합니다. 가드가 없으면 이미 패칭된 함수를 또 패칭하게 되어, 인덱스가 한 번에 2씩 증가하는 등 추적이 망가집니다. "Once"라는 이름이 여기서 보장됩니다.

② 원본 메서드 백업

139const originalHistoryPushState = window.history.pushState;
140const originalHistoryReplaceState = window.history.replaceState;
목적
패칭하기 전에 브라우저의 진짜 함수를 변수에 보관해 둔다.
이유
패치도 결국 "실제로 history를 바꾸는" 일은 원본에 위임해야 합니다(original.call(...)). 백업해 두지 않으면 패치가 자기 자신을 부르는 무한 재귀에 빠지거나 진짜 동작이 사라집니다. DEBUG일 때는 원본을 window.__next_navigation_debug에도 노출합니다(:142-147).

③ 캡처된 세션 복원, 없으면 fresh 시작

149// 캡처된 세션(새로고침)이 있으면 복원, 없으면(진짜 첫 방문) fresh 시작.
150if (_capturedSessionStateAtModuleLoad) {
151  setRenderedState(_capturedSessionStateAtModuleLoad);                       // ← 복원
152} else {
153  setRenderedState({ historyIndex: 0, sessionToken: generateSessionToken() }); // ← 첫 방문
154}
목적
모듈 포인터를 이전 세션이 있으면 그 값으로 복원하고, 없으면 { historyIndex: 0, 새 토큰 }으로 초기화한다. 새로고침이면 원래 세션에 재합류하고, 진짜 첫 방문이면 "내 세션의 0번 진입점"으로 출발한다.
이유
이전 세션의 인덱스/토큰은 모듈 평가 시점에 이미 캡처해 두었습니다 (전용 절). 그래서 새로고침이라도 과거를 버리지 않고 그대로 복원해, back/forward를 인덱스(delta)로 정상 구분합니다. 이 시점까지 _renderedState는 무효값 { -1, "" }이었고, 여기서 비로소 유효해집니다.

④ 상태 갱신 함수 "정의" (아직 실행 아님)

159_setRenderedStateAndSyncToHistory = ({ renderedState, shouldSyncToHistory }) => {
160  debug(…);
164  setRenderedState(renderedState);                  // 1) 항상 모듈 포인터 갱신
165
166  if (shouldSyncToHistory) {                          // 2) 필요할 때만 history.state 반영
167    originalHistoryReplaceState.call(
168      window.history,
169      withTrackingMetadata(window.history.state, renderedState),
170      "",
171      window.location.href                            // URL은 그대로, state만 교체
172    );
173  }
174};
주의 — 이 단계는 "선언"일 뿐, 여기서 상태가 바뀌지 않습니다. 이 코드는 모듈 변수 _setRenderedStateAndSyncToHistory에 함수(클로저)를 할당할 뿐입니다. 본문은 나중에 이 함수가 호출될 때 비로소 실행됩니다. 실제 첫 실행은 ⑤(시드)이고, 이후 popstate 핸들러가 통과 분기에서 호출합니다.
목적
"우리 추적 상태(포인터)"와 "브라우저의 history.state(메타데이터)"를 동기화하는 단일 창구를 미리 정의해 둔다.
구현
호출되면 — 항상 setRenderedState로 포인터를 갱신하고, shouldSyncToHistory: true일 때만 추가로 원본 replaceState로 현재 진입점의 history.state에 우리 메타데이터를 덮어씁니다. URL은 그대로 둡니다. 이 클로저는 rendered-state-context.ts가 setState(sync OFF) 한 이름으로만 감싸 노출합니다. sync ON 경로는 외부로 노출되지 않고, initialize의 ⑤ 시드가 이 클로저를 shouldSyncToHistory: true로 직접 한 번 부르는 게 전부입니다(과거의 setStateAndSyncToHistory 래퍼는 경계 leave 경로가 pendingRestore로 병합되며 제거됨).

⑤ 현재 진입점에 메타데이터 시드 (복원 세션 재-stamp 포함)

183if (
184  !window.history.state ||
185  window.history.state[STACK_INDEX_KEY] == null ||
186  window.history.state[SESSION_TOKEN_KEY] == null
187) {

목적
지금 머무는 진입점에 우리 메타데이터가 아직 없으면, 즉시 심어준다. 첫 방문이면 fresh { index:0, token }을, 새로고침이면 ③에서 복원한 세션을 그 진입점에 다시 stamp한다.
이유
패치(⑥)는 앞으로 일어날 push/replace에만 끼어듭니다. 하지만 사용자가 지금 서 있는 현재 진입점은 패치 설치 전에 이미 만들어져 push를 거친 적이 없습니다 → 메타데이터가 비었습니다. 게다가 새로고침의 경우 복원한 토큰은 메모리에만 살아 있고, 디스크의 그 진입점은 (Next가 hydration 때 덮어써서) 여전히 token-less입니다. 그래서 여기서 한 번 수동 주입해 "패치가 영원히 닿지 못하는 0번 진입점"을 부트스트랩합니다.
[진입점 0] ← 이미 존재. 패치 이전 생성. 메타데이터 ✗ │ (여기서 initialize → 패치 설치) │ ⑤ 분기가 0번에 메타데이터를 손으로 심음 ✓ ▼ router.push() ─► 패치된 pushState ─► [진입점 1] 메타데이터 자동 기록 ✓ ▼ router.push() ─► 패치된 pushState ─► [진입점 2] 메타데이터 자동 기록 ✓

세 조건은 왜 OR인가

세 검사는 ||로 묶여 하나라도 참이면 주입합니다. == null은 null과 undefined를 모두 잡습니다(키가 없으면 값은 undefined).

조건참이 되는 때구체 시나리오
!window.history.state history.state가 null URL 직접 입력 · 외부 링크 · 새 탭/북마크 — 누구도 pushState(state)를 부른 적 없는 순수 첫 진입점

목적
같은 팩토리(④절)에 원본 + advanceIndex 플래그만 달리 넣어 두 메서드를 교체한다. true는 인덱스 +1(새 진입점), false는 인덱스 유지(덮어쓰기).
이유
pushState는 스택에 한 칸 더 쌓지만, replaceState는 같은 칸을 덮어씁니다. 이 차이를 정확히 반영해야 delta 계산이 어긋나지 않습니다. 또 Next가 라우팅 중 replaceState로 state를 갈아끼울 때 우리 메타데이터가 날아가지 않도록, replaceState도 반드시 패칭해 매번 다시 심습니다. 마지막으로 플래그를 켜고(①의 짝), ④의 sync 함수를 호출자(createRenderedStateContext)에게 돌려줍니다.

7. 패칭의 결과는 언제, 어떻게 쓰이나

몽키패칭은 그 자체로는 아무 일도 하지 않습니다. 단지 모든 history 진입점에 메타데이터를 기록해두는 준비 작업입니다. 진짜 소비는 사용자가 뒤로/앞으로 가기를 눌러 popstate가 발생할 때 useInterceptPopState.ts에서 일어납니다.

1
popstate 발생 → history.state를 parse-history-state.ts의 parseHistoryState()에 넘김 → 패치가 심어둔 __next_navigation_stack_index(=nextHistoryIndex)와 __next_session_token(=nextSessionToken)을 꺼낸다.
2
[Confirmed] 검사 — 직전에 핸들러가 승인해 우리가 부른 go/back이면 그대로 통과(handleConfirmedNavigation).
3
토큰 비교: hasSessionTokenMismatch(next, current). 다르면 → 진짜 세션 경계(handleSessionBoundary): 토큰 없는 진입점이거나 외래 토큰. 인덱스를 비교할 수 없어 "뒤로"로 가정하고 history.go(1)로 URL 복원. 새로고침은 토큰이 모듈-평가에서 복원되어 여기 안 옴 — 아래 delta 분기로 처리됩니다.
4
delta 계산(handleInternalNavigation): delta = nextHistoryIndex − currentRenderedState.historyIndex. 이 한 줄이 몽키패칭의 존재 이유입니다.
delta < 0 뒤로 → 가드. go(-delta)로 복원 후 핸들러. delta > 0 앞으로 → 통과. delta = 0 복원 완료 신호 → 보류 핸들러 실행.
5
통과 분기에선 대부분 setState로 포인터만 갱신(도착지 메타데이터가 이미 정확). history.state까지 다시 쓰는 sync는 비었거나 신뢰 불가일 때만 일어난다. 루프가 닫힌다.
요점: initialize/패치는 "데이터를 심는 쪽(write)", popstate 핸들러는 "그 데이터를 읽어 방향을 판단하는 쪽(read)"입니다. 둘은 모듈 싱글톤 _renderedState와 history.state를 공유 채널로 씁니다.

9. 한눈에 요약

질문답
무엇을 몽키패칭하나?window.history.pushState 와 window.history.replaceState
어떻게 패칭하나?단일 팩토리 createPatchedHistoryMethod(원본, advanceIndex) — push는 true, replace는 false
왜 하나?History API에 "현재 위치 인덱스"가 없어 popstate의 방향(뒤/앞)을 알 수 없음. Navigation API는 Safari/Firefox 미지원.
무엇을 심나?__next_navigation_stack_index(위치) + __next_session_token(세션 ID) — withTrackingMetadata가 병합
push 패치 동작인덱스 +1 후 메타데이터 주입 → 원본 호출 (새 진입점)
replace 패치 동작인덱스 유지, 메타데이터만 재주입 → 원본 호출 (덮어쓰기)
왜 "Once"인가?중복 패칭 시 인덱스가 2씩 증가해 추적이 깨짐. _isHistoryStateSyncInitialized 플래그로 1회만.
토큰은 어떻게 정하나?모듈-평가 시점에 이전 토큰을 캡처해 복원(_capturedSessionStateAtModuleLoad), 없을 때만(첫 방문) generateSessionToken(). 새로고침은 재합류 → mismatch 아님.
현재 진입점(0번)은?패치가 닿지 못하므로 ⑤에서 수동 시드. 첫 방문은 fresh, 새로고침은 복원 세션 재-stamp.
sync ON은 언제?도착 칸 메타데이터가 비었을 때 — ⑤ 시드 단 1곳뿐. 나머지 통과 분기는 도착 칸이 이미 정확해 멱등이라 OFF. (경계 leave의 옛 sync ON은 pendingRestore 병합으로 제거됨.)
심은 데이터는 언제 쓰이나?popstate 시 useInterceptPopState.ts가 읽어 delta 계산 + 토큰 비교 → 뒤로 가기만 가드.
