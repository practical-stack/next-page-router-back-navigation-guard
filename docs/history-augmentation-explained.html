History augmentation — 현재 구현 해설

history-augmentation.ts와 rendered-history-entry-metadata-store.ts가 브라우저 history entry와 현재 렌더링 화면의 위치를 어떻게 동기화하는지 설명한다.

왜 자체 metadata가 필요한가

History API는 현재 stack index를 노출하지 않는다. 브라우저는 popstate callback보다 먼저 활성 entry, URL, history.state를 바꾸므로 callback 시점의 history.state만으로는 이동 전 화면의 위치도 알 수 없다.

해결책: 모든 앱 entry에 session token과 stack index를 기록하고, 현재 렌더링된 entry의 metadata는 메모리에 별도로 유지한다.
{
  __next_session_token: string,
  __next_navigation_stack_index: number
}

현재 모듈 상태

const _capturedRenderedHistoryEntryMetadataAtModuleLoad = /* history.state */;

let _isHistoryStateSyncInitialized = false;
let _renderedHistoryEntryMetadata = {
  historyIndex: -1,
  sessionToken: "",
};
  • _capturedRenderedHistoryEntryMetadataAtModuleLoad: hydration 전에 이전 token/index를 캡처한다.
  • _isHistoryStateSyncInitialized: History API patch를 한 번만 수행한다.
  • _renderedHistoryEntryMetadata: 현재 화면이 나타내는 entry의 token/index다.

이전 구현의 _setRenderedStateAndSyncToHistory closure와 rendered-state-context.ts는 존재하지 않는다. 현재는 metadata setter와 전용 store가 역할을 나눈다.

새로고침 복구

브라우저는 reload 동안 현재 entry의 history.state를 보존하지만 Next.js hydration이 나중에 이를 덮어쓴다. Provider mount 시점은 늦기 때문에 파일의 module evaluation 시점에 token과 index를 먼저 캡처한다.

const captured = history.state?.__next_session_token
  ? {
      sessionToken: history.state.__next_session_token,
      historyIndex: Number(history.state.__next_navigation_stack_index) || 0,
    }
  : null;

initializeHistoryStateSyncOnce()는 캡처 값이 있으면 복원하고, 없을 때만 index 0과 새 token으로 새 session을 시작한다. 따라서 정상 reload는 session boundary가 아니다.

원본 History method가 성공한 뒤 메모리 포인터를 갱신한다. 전달된 state가 primitive이면 metadata를 병합할 수 없으므로 빈 object를 기반으로 새 state를 만든다.

초기화와 현재 entry stamp

  1. 원본 pushState/replaceState를 저장한다.
  2. 캡처한 metadata를 복원하거나 새 session을 만든다.
  3. 현재 entry에 metadata가 없으면 원본 replaceState로 stamp한다.
  4. 두 History method를 patch한다.
  5. 초기화 flag를 설정하고 setRenderedHistoryEntryMetadata를 반환한다.

현재 entry stamp만 history.state를 새로 쓴다. popstate로 이미 존재하는 entry에 도착했을 때는 해당 entry에 metadata가 기록돼 있으므로 메모리 포인터만 이동한다.

Rendered metadata store

const store = createRenderedHistoryEntryMetadataStore();

store.get(); // immutable metadata copy
store.set(nextRenderedHistoryEntryMetadata); // memory only

createRenderedHistoryEntryMetadata()는 confirmed navigation으로 도착한 entry의 metadata를 만든다. token이 없으면 새 session을 시작하고 index를 0으로 둔다. token이 있으면 그 token과 entry index를 사용한다.

moveRenderedHistoryEntryMetadataToIndex()는 같은 session 내부 forward 또는 handler 없는 back을 허용할 때 token을 보존하고 index만 옮긴다.

popstate interceptor와의 연결

앱 초기화 → createPopstateInterceptor() → createRenderedHistoryEntryMetadataStore() → initializeHistoryStateSyncOnce() popstate 도착 → 도착 entry의 token/index parse → store.get()의 렌더링 entry와 비교 → session boundary / forward / back 분류 → navigation 허용 시 store.set()으로 렌더링 포인터 이동

뒤로가기를 차단하고 URL을 복원하는 동안에는 store를 이동하지 않는다. 화면은 여전히 원래 entry를 렌더링하기 때문이다. handler가 승인한 replay popstate 또는 즉시 허용된 navigation에서만 도착 entry로 이동한다.
