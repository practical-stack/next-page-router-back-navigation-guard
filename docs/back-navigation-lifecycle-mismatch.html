정리

복원 echo는 어떻게 처리되나 빈 handlerMap fast path 정상 back과 한눈 대비 전체 타임라인 한 장 상태 변화 표 ← 정상 back navigation 생애주기 문서

뒤로가기 한 번의 생애주기 — 세션 경계(토큰 mismatch)

이 문서는 src/useInterceptPopState.helper/popstate-interceptor.ts의 session boundary 분기 — handleSessionBoundary가 타는 흐름을 시간순 tick 단위로 추적합니다. 토큰이 일치하는 평범한 내부 뒤로가기는 back-navigation-lifecycle.html에서 다룹니다. 이 문서는 토큰이 어긋났을 때(mismatch)만 다룹니다.

범위: 토큰 mismatch · 핸들러 1개 등록됨 한 케이스만 끝까지 추적합니다. 정상 back(토큰 일치)은 자매 문서에서 다룹니다.

0. 이 문서가 추적하는 것

정상 back과 mismatch는 "뒤로가기를 일단 되돌려 놓고 → 핸들러에게 물어보고 → 승인되면 다시 한다"는 큰 줄기가 같습니다. 사실 완료 감지·핸들러 실행·재실행까지 모두 같은 pendingNavigation 메커니즘을 공유합니다. 유일하게 다른 건 "되돌리는 방법"입니다. 정상 back은 인덱스 델타로 정확히 go(-delta)하지만, 경계 너머에서는 인덱스가 서로 다른 세션의 것이라 비교 불가라 델타를 못 구합니다 → "한 칸 뒤" 가정.

경계는 정상 back과 거의 같은 길을 탑니다. 실질적 차이는 "되돌리는 방법" 하나뿐입니다.

  • 복원(유일한 차이): 델타를 못 구하므로 "한 칸 뒤로 갔다고 가정"하고 begin(-1) + history.go(1)로 되돌린다. (정상 back은 begin(delta) + go(-delta) — 정확한 칸수.)
  • 완료 감지(정상 back과 동일): begin(-1) 후 go(1)하면 그 echo가 delta===0으로 돌아오고, 토큰이 일치하는 엔트리(B)라 handleInternalNavigation의 delta===0 && hasPending() 분기가 completeRestoreAndRunHandlers를 부른다. 새로고침처럼 echo가 안 올 때를 위해 scheduleRestoreCompletionFallback(rAF+setTimeout) fallback도 같이 건다 — 둘 중 먼저 오는 쪽이 이긴다(consume 멱등).

mismatch란 무엇인가

뒤로가기로 도착한 엔트리의 history.state에 담긴 세션 토큰이 현재 세션 토큰과 다를 때입니다. 판정은 단 한 줄(parse-history-state.ts):

function hasSessionTokenMismatch(nextSessionToken, currentSessionToken) {
  return !nextSessionToken /* 토큰 없음 */ || nextSessionToken !== currentSessionToken /* 다른 토큰 */;
}

추적할 시나리오 — 가드를 부분 적용한 앱

가드는 페이지 B에만 적용돼 있다(점진적 도입). 사용자는 가드가 없는 레거시 페이지 A에서 B로 넘어왔다. A는 우리 pushState 패치가 깔리기 전에 push됐으므로 메타데이터가 전혀 없다(∅). B에 도착해 Provider가 마운트되며 패치가 깔리고 현재 엔트리 B가 stamp된다. 이제 B에서 사용자가 뒤로가기를 누른다.

history 스택: A { state: ∅ — 토큰 없음 } → B {idx:0, token:τ} ← 현재

τ = 현재 세션 토큰. ∅ = 메타데이터 없음 → 뒤로가기로 A에 도착하면 토큰 mismatch. renderedMetadata(라이브러리의 "진실") = { historyIndex: 0, sessionToken: τ }

준비 단계 — 토큰 없는 A는 어떻게 생기나

P1
Browser A는 패치 이전에 push됨 — 메타데이터 없음 history-augmentation.ts

가드 Provider가 B에서만 마운트되므로, A로 이동하던 시점엔 history.pushState가 아직 패치되지 않은 원본이었다. 그래서 A의 history.state엔 우리 키 (__next_session_token / __next_navigation_stack_index)가 없다.

※ 같은 결과를 내는 다른 경로: 외부 사이트에서 B로 진입, 라이브러리 도입 전 만들어진 엔트리 등 — 공통점은 "우리 토큰이 없다".

A { state: ∅ } ← 현재(아직 B 이동 전)
P2
Library A → B 이동 후 B 마운트 → 패치 + 현재 엔트리 stamp

A→B 이동(router.push("/b")) 시점엔 패치가 없어 B도 메타데이터 없이 push된다. 그 직후 B에서 Provider가 마운트되며 initializeHistoryStateSyncOnce()가:

  • 모듈-평가 캡처 없음(새 진입) → renderedMetadata = { idx:0, token:τ(새로 생성) }
  • history.pushState / replaceState 몽키패치
  • 현재 엔트리 B에 stamp → B.state = { idx:0, token:τ }

beforePopState도 등록 → 이후 popstate마다 우리 핸들러가 불린다.

A { ∅ } → B {idx:0, τ} ← 현재 · 핸들러 등록됨
renderedMetadata{ idx: 0, token: τ } history.state (B){ idx: 0, token: τ } history.state (A)∅ (메타데이터 없음) nextNavigationConfirmedfalse
준비 끝. B에 있고, renderedMetadata/B.state는 {idx:0, τ}로 일치하지만 그 뒤의 A에는 토큰이 없다. 여기서 사용자가 뒤로가기를 누르면 — A에 도착하는 순간 mismatch가 터진다.

경계 뒤로가기 연쇄 — 토큰이 어긋난 채로 도착

T1
User 뒤로가기 버튼 누름

사용자가 뒤로가기(또는 router.back())를 한 번 누른다. 브라우저는 즉시 스택 포인터를 A로 옮긴다 — 막을 수 없다. URL 바는 A를 가리키고, history.state는 A의 것 ∅(토큰 없음)으로 바뀐다.

A { ∅ } ← 지금 여기로 끌려옴 → B {idx:0, τ}
T2
popstate 1차 popstate → handleSessionBoundary → block

이동 완료로 popstate 발생 → interceptPopstate(history.state). 인자는 A의 state ∅.

// 파싱: nextSessionToken = undefined, nextHistoryIndex = Number(undefined)||0 = 0
// current(renderedMetadata) = { idx:0, τ }

// 분기 0: nextNavigationConfirmed? false → skip
// 분기 1: hasSessionTokenMismatch(undefined, τ)? true (!undefined) → 여기!

토큰이 없으니 mismatch → handleSessionBoundary 진입. 핸들러가 등록돼 있으므로(hasRegisteredHandlers true) 빈-맵 fast path를 건너뛴다:

pendingNavigation.begin(-1);    // 델타 못 구함 → "한 칸 뒤" 가정
window.history.go(1);                // -(-1) = +1 → B로 복원
scheduleRestoreCompletionFallback(fallback);  // echo가 안 올 때(새로고침) 대비 보험
return false;                  // ★ Next.js 라우팅 차단
왜 go(-delta)가 아니라 go(1)인가: 경계 너머 A의 인덱스(0)는 우리 세션 τ의 인덱스가 아니다. 서로 다른 세션의 숫자를 빼봐야 의미가 없으므로 델타를 포기하고, "뒤로가기 = 한 칸 뒤"라는 가정으로 go(1) 복원한다.
정상 back과 같은 pendingNavigation를 쓴다: begin(-1)만 넣어 두면 go(1) echo가 delta===0으로 돌아올 때(T4) handleInternalNavigation의 delta===0 && hasPending() 분기가 completeRestoreAndRunHandlers를 불러 — 정상 back과 완전히 같은 완료 경로가 된다. scheduleRestoreCompletionFallback는 "echo가 안 오는 새로고침"용 fallback일 뿐이다(consume 멱등 → 둘 중 먼저 온 쪽만 실행).
복원 echo가 곧 완료 신호: go(1) 복원은 토큰이 일치하는 엔트리(B)로 되돌아오므로, 그 echo는 경계가 아니라 handleInternalNavigation으로 떨어진다. 거기서 delta===0 && hasPending()이라 echo가 그대로 핸들러 실행을 발화한다(T4). 별도 플래그 없이 "토큰 일치 + pending" 두 조건만으로 동작한다(아래 절).
renderedMetadata{ idx:0, τ } (안 바뀜) pendingNavigation{ delta: -1 } 반환false → Next 차단
T3
Browser go(1) 실행 → URL을 B로 복원

브라우저가 go(1)을 비동기 처리해 스택 포인터를 다시 B로 옮긴다. URL 바가 B로 돌아온다 — 사용자 눈엔 아직 "뒤로가기가 안 먹힌 것"처럼 보인다(핸들러를 아직 안 물어봤으니 당연).

A { ∅ } → B {idx:0, τ} ← 복원됨
T4
popstate 2차 popstate(복원 echo) → 토큰 일치 → delta===0 + pending → 핸들러 발화

go(1) 완료로 popstate 재발생. 인자는 이제 B의 state {idx:0, τ} — 토큰이 τ로 일치한다. 그래서 이 echo는 분기 1(경계)이 아니라 분기 2(Internal)로 떨어진다.

// nextSessionToken = τ, nextHistoryIndex = 0, current.idx = 0
// 분기 1: hasSessionTokenMismatch(τ, τ)? false → skip → handleInternalNavigation
historyIndexDelta = 0 − 0 = 0;

if (delta === 0 && pendingNavigation.hasPending()) {   // ← T2에서 begin(-1) 했음!
  completeRestoreAndRunHandlers();                          // 드디어 핸들러 실행 (정상 back과 동일)
  return false;
}
핵심: 경계가 일으킨 go(1) 복원의 echo popstate가 완료 신호 역할을 한다. 복원 destination B의 토큰이 τ로 일치해 echo가 Internal로 라우팅되고, delta===0 && hasPending()이라 completeRestoreAndRunHandlers가 불린다 — 정상 back의 T4와 토씨까지 같은 경로다. (새로고침이라 이 echo가 안 오면 T2의 scheduleRestoreCompletionFallback fallback이 대신 부른다.)
renderedMetadata{ idx:0, τ } (안 바뀜) pendingNavigationconsume → null 반환false
T5
Library completeRestoreAndRunHandlers → 핸들러에게 물어봄

T4의 echo가 부른 completeRestoreAndRunHandlers가 보류해 둔 복원을 consume하고 핸들러를 실행한다(새로고침이면 T2의 fallback이 같은 함수를 부른다). 정상 back과 완전히 같은 함수·같은 흐름이다.

const consumedNavigation = pendingNavigation.consume();  // {delta:-1} 꺼내고 null로 (원자적)
if (!consumedNavigation) return;                     // 이미 소비됐으면 no-op (멱등)

왜 nextNavigationConfirmed = true를 먼저 세우나: 곧 부를 go(-1)이 또 popstate를 일으킨다. 이걸 새 경계로 오인하면 핸들러를 또 실행 → 또 승인 → 또 go() → 무한 루프. 이 표식이 분기 0에서 핸들러를 건너뛰게 해 한 번에 끝낸다. (정상 back의 분기 B와 완전히 동일 — go(consumedNavigation.delta)가 여기선 go(-1)일 뿐이다.)

※ leave 재실행 단계는 renderedMetadata를 건드리지 않는다(completeRestoreAndRunHandlers는 nextNavigationConfirmed만 세우고 go(-1)). 새 세션 토큰은 T7 도착 시점에 한 번만 생긴다(토큰 없는 ∅ → createRenderedHistoryEntryMetadata가 새 토큰 τ″ 생성). 토큰이 있는(σ) 경우엔 그 세션으로 그대로 전환된다.

곁가지 ① — 복원 echo는 어떻게(플래그 없이) 처리되나

handleSessionBoundary의 go(1) 복원이 일으키는 echo popstate는 복원이 도착하는 엔트리의 토큰과 pending 유무에 따라 둘 중 하나로 처리된다 — 어느 쪽이든 전용 플래그는 없다.

핸들러 등록됨 + begin(-1) 후 (이 문서 주 시나리오)handleInternalNavigation의 delta===0 && hasPending()completeRestoreAndRunHandlers 발화 = 완료 신호 (T4)
빈 handlerMap에서 block (곁가지 ②)handleInternalNavigation의 delta===0 (pending 없음)return false로 삼킴
공통 원리: go(1) 복원은 항상 우리가 출발했던 현재 엔트리(B)로 되돌아오고, B는 현재 토큰 τ를 가지므로 echo는 경계(mismatch)가 아니라 handleInternalNavigation으로 라우팅된다. 거기서 pending이 있으면 완료 신호로 쓰이고(핸들러 발화), 없으면(빈-맵 block) 이동이 없는 delta===0 echo라 그냥 무시된다. "복원 중"임을 기억하는 별도 일회성 플래그는 어느 경우에도 필요 없다.

곁가지 ② — 등록된 핸들러가 없을 때 (빈 handlerMap fast path)

handlerMap이 비어 있으면(실사용 핸들러 0개) preRegisteredHandler(예: 오버레이 닫기)만 남는다. 경계 분기는 이걸 보류 없이 즉시 부른다.

if (!hasRegisteredHandlers(handlerMap)) {
  if (preRegisteredHandler && !preRegisteredHandler()) {  // 부수효과(오버레이 닫힘) 후 block
    window.history.go(1);   // URL 복원하고 멈춤
    return false;
  }
  renderedHistoryEntryMetadataStore.set(createRenderedHistoryEntryMetadata({ sessionToken: nextSessionToken, historyIndex: nextHistoryIndex }));  // 도착 엔트리를 새 기준으로
  return true;   // 통과
}
block 시 URL을 반드시 복원하는 이유: 복원하지 않으면 브라우저 URL은 이전 페이지로 가 있는데 Next.js는 현재 페이지를 렌더하는 어긋남이 생긴다. 최악(once 핸들러 + 경계): 1차 back이 막으며 핸들러 삭제 → 2차 back이 모달 닫음 → URL 미복원이면 이미 홈에 있어 3차 back이 사이트를 떠난다(about:blank). 그래서 막을 땐 go(1)로 되돌리는 게 필수.

한 줄 요약. 토큰이 어긋난 채 도착 = ① 막을 수 없는 강제 이동 → ② 1차 popstate에서 델타 대신 begin(-1) + go(1)로 되돌린 뒤 차단 → ③ 복원 echo가 토큰 일치 엔트리(B)로 돌아와 Internal의 delta===0 + pending 분기에서 completeRestoreAndRunHandlers를 발화 → ④ 핸들러에게 물어봄 → ④-false면 머무름 / ④-true면 nextNavigationConfirmed 세우고 go(-1) → ⑤ 3차 popstate를 분기 0이 통과시켜 새 세션으로 갈아타며 이동 완료. 정상 back과 완전히 같은 pendingNavigation 경로를 공유하고, 다른 건 "인덱스를 못 믿으니 델타 대신 한 칸 가정(go(1))"과 "떠난 뒤 새 세션 전환"뿐이다.
