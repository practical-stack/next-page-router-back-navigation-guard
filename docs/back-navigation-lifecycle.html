뒤로가기 한 번의 생애주기 — 정상 back navigation

이 문서는 src/useInterceptPopState.helper/popstate-interceptor.ts에서 사용자가 뒤로가기 버튼을 한 번 누른 순간부터 이동이 완료(또는 차단)될 때까지 일어나는 일을 시간순 tick 단위로 추적합니다. 코드 구조 해설은 popstate-interceptor.ts를 보세요 — 이 문서는 "흐름"만 다룹니다.

범위: 가장 기본인 일반(내부) 뒤로가기 — 토큰 일치 · 새로고침 아님 · 핸들러 1개 등록됨 한 케이스만 끝까지 추적합니다. (세션 경계 / 새로고침 직후 / 핸들러 없음은 다른 분기이며 본문 끝에서 짧게 연결합니다.)

0. 이 문서가 추적하는 것

핵심 난점은 "뒤로가기는 이미 일어난 뒤에 popstate로 통보된다"는 점입니다. 취소할 방법이 없으니, 라이브러리는 일단 일어난 이동을 되돌려(URL 복원) 놓고 → 핸들러에게 물어본 뒤 → 승인되면 그 이동을 다시 직접 재실행합니다. 그래서 사용자의 단 한 번의 뒤로가기가 내부적으로는 popstate를 2~3번 발생시킵니다.

추적할 시나리오

페이지 A → B로 이동(router.push)해 현재 B에 있고, B에는 "저장 안 됨" 가드 핸들러가 1개 등록되어 있다. 여기서 사용자가 뒤로가기를 누른다.

history 스택: A {idx:0, token:τ} → B {idx:1, token:τ} ← 현재

τ = 세션 토큰(이 세션 내내 동일). renderedMetadata(라이브러리의 "진실") = { historyIndex: 1, sessionToken: τ }

준비 단계 — 뒤로가기를 누르기 전에 이미 끝나 있어야 하는 것

뒤로가기를 추적하기 전에, 그 시점의 상태가 어떻게 만들어졌는지부터 봅니다. 이 3단계가 "메타데이터는 언제 박히나"에 대한 답입니다.

P0
Browser 모듈 평가 — 이전 세션 토큰 캡처 history-augmentation.ts

스크립트 번들이 평가되는 순간, Next 라우터가 hydration으로 history.state를 덮어쓰기 전에 한 번 읽어 둔다. 새로고침이라면 이전 토큰이 아직 남아 있어 캡처되고, 진짜 첫 방문이면 null.

const _capturedRenderedHistoryEntryMetadataAtModuleLoad =
  window.history.state?.[SESSION_TOKEN_KEY]
    ? { sessionToken: …, historyIndex: … }   // 새로고침 → 복원용
    : null;                                // 첫 방문

※ 이번 시나리오는 "첫 방문 후 A→B 이동"이라 null. 새로고침 케이스는 맨 끝 참고.

P1
Library Provider 마운트 → 패치 + beforePopState 등록

useInterceptPopState의 useIsomorphicLayoutEffect가 실행되며 두 가지를 한다.

① createPopstateInterceptor() → 내부에서 createRenderedHistoryEntryMetadataStore() → initializeHistoryStateSyncOnce() 호출:

  • 캡처된 세션 없음 → renderedMetadata = { idx:0, token:τ(새로 생성) }
  • history.pushState / replaceState를 몽키패치 (이후 모든 이동에 메타데이터 자동 주입)
  • 현재 엔트리 A에 메타데이터 stamp → A.state = { idx:0, token:τ }

② beforePopState 등록 — popstate가 올 때마다 우리 핸들러가 불리도록 연결:

pagesRouter.beforePopState(() => interceptPopstate(window.history.state));
A {idx:0, token:τ} ← 현재
P2
User A → B 이동 (router.push) — 여기서 B의 메타데이터가 박힌다

Next의 router.push("/b")는 내부적으로 패치된 history.pushState를 호출한다. 패치 함수가 원본 호출 직전에 끼어들어:

// createPatchedHistoryMethod (advanceIndex = true)
const next = { sessionToken: τ, historyIndex: 0 + 1 };  // 인덱스 +1
setRenderedHistoryEntryMetadata(next);                                  // 메모리: renderedMetadata = {idx:1, τ}
originalPushState(withTrackingMetadata(state, next), …);    // B.state = {idx:1, τ}
A {idx:0, τ} → B {idx:1, τ} ← 현재
이 문서가 답하려는 첫 질문: "router.push의 pushState 시점에 메타데이터 생성" — 바로 여기다. 라이브러리는 push를 가로채는 게 아니라, push가 부르는 pushState에 메타데이터를 끼워 넣을 뿐 이동 자체는 그대로 통과시킨다(뒤로가기만 가드).
renderedMetadata{ idx: 1, token: τ } history.state (B){ idx: 1, token: τ } nextNavigationConfirmedfalse pendingNavigationnull
준비 끝. 이제 무대 위 상태: B에 있고, renderedMetadata와 B의 history.state가 둘 다 {idx:1, τ}로 일치, 플래그는 모두 초기값. 여기서 사용자가 뒤로가기를 누른다.

뒤로가기 연쇄 — 한 번의 입력이 만드는 popstate들

T1
User 뒤로가기 버튼 누름

사용자가 브라우저 뒤로가기(또는 router.back())를 한 번 누른다. 이게 사용자의 마지막 능동적 행동이고, 이후는 전부 자동 연쇄다.

브라우저는 즉시 스택 포인터를 A로 옮긴다 — 이건 막을 수 없다. URL 바는 벌써 A를 가리키고, history.state도 A의 것 {idx:0, τ}으로 바뀐다.

A {idx:0, τ} ← 지금 여기로 끌려옴 → B {idx:1, τ}
T2
popstate 1차 popstate → Library 판정 → block

이동이 끝났으니 popstate 발생 → Next가 가로채 interceptPopstate(history.state) 호출. 인자는 A의 state {idx:0, τ}.

3개 분기 판정:

// 파싱: nextSessionToken = τ, nextHistoryIndex = 0
// current(renderedMetadata) = { idx:1, τ }

// 분기 0: nextNavigationConfirmed? false → skip
// 분기 1: hasSessionTokenMismatch(τ, τ)? false(토큰 일치) → skip
// 분기 2: handleInternalNavigation
historyIndexDelta = 0 − 1 = -1   // 음수 = back!

delta가 음수 → back 가드 발동. 핸들러가 등록돼 있으므로(hasRegisteredHandlers true) 빠른 경로를 건너뛰고 핵심 로직 실행:

pendingNavigation.begin(-1);   // "복원 끝나면 delta -1을 재생하라" 기억
window.history.go(1);         // -(-1) = +1 → B로 되돌림 (URL 복원)
scheduleRestoreCompletionFallback(fallback);  // 새로고침-안전 보험 예약
return false;             // ★ Next.js 라우팅 차단
여기서 핸들러를 바로 부르지 않는 이유: history.go(1)은 비동기다. 복원이 끝나기 전에 핸들러가 router.push()를 호출하면 스택이 꼬인다. 그래서 핸들러를 pendingNavigation에 보류해 두고, 복원 완료(다음 popstate)를 기다린다.
renderedMetadata{ idx: 1, τ } (안 바뀜) pendingNavigation{ delta: -1 } 반환false → Next 차단
T3
Browser go(1) 실행 → URL을 B로 복원

브라우저가 go(1)을 비동기 처리해 스택 포인터를 다시 B로 옮긴다. URL 바가 B로 돌아온다 — 사용자 눈엔 "뒤로가기가 안 먹힌 것"처럼 보인다(아직 핸들러를 안 물어봤으니 당연).

A {idx:0, τ} → B {idx:1, τ} ← 복원됨
T4
popstate 2차 popstate → delta === 0 → "복원 완료" 신호

go(1)이 끝나 popstate 재발생. 인자는 이제 B의 state {idx:1, τ}.

// nextHistoryIndex = 1, current.idx = 1 (둘 다 안 바뀜)
historyIndexDelta = 1 − 1 = 0;

if (historyIndexDelta === 0) {
  if (pendingNavigation.hasPending()) {       // ← 그렇다! T2에서 begin(-1) 했음
    completeRestoreAndRunHandlers();             // 드디어 핸들러 실행
    return false;
  }
}
delta === 0의 의미: "방금 우리가 부른 go()로 인한 복원이 완료됐다"는 신호. MDN이 권장하는 "history.go 완료 감지 = popstate 대기" 패턴 그대로다. pending이 걸려 있으니 이 popstate가 바로 "이제 핸들러를 돌려도 안전하다"는 출발 신호.
T5
Library completeRestoreAndRunHandlers() → 핸들러에게 물어봄
const consumedNavigation = pendingNavigation.consume();  // {delta:-1} 꺼내고 null로 비움(원자적)
if (!consumedNavigation) return;                     // 이미 소비됐으면 no-op

void (async () => {
  const destinationPath = location.pathname + location.search;
  const shouldAllow =
    await runHandlerChain({ handlerMap, preRegisteredHandler, destinationPath });
  …  // 분기 A / B (아래)
})();

consume()가 읽으면서 동시에 비우므로, 뒤이어 fallback이 실행돼도 consume()은 null → 자동 no-op. "정확히 한 번"이 보장된다.

핸들러 체인은 우선순위 1등 하나만 await로 실행하고 그 boolean을 받는다. 사용자에게 "정말 나가시겠어요?" 다이얼로그를 띄우는 게 보통 여기다. 결과에 따라 두 갈래로 나뉜다.

분기 — 핸들러의 대답

분기 A · false 반환 → 머무름 (가드 성공)

사용자가 다이얼로그에서 "취소"를 눌렀다 → 핸들러가 false.

if (!shouldAllow) {
  debug("[Internal] Handler blocked");
  return;   // 그냥 끝. 아무 history 조작도 안 함
}
A {idx:0, τ}→B {idx:1, τ} ← 그대로 머무름
결과: URL은 T3에서 이미 B로 복원됐고, 핸들러가 막았으니 더 할 일이 없다. 사용자는 B에 그대로. 뒤로가기가 깔끔히 차단됨 — 이게 라이브러리의 존재 이유. 플래그/pending 모두 초기 상태로 돌아가, 다음 뒤로가기는 처음부터 다시 가드된다. 여기서 종료 · popstate는 더 안 남

분기 B · true 반환 → 떠남 (이동 직접 재실행)

사용자가 "나가기"를 눌렀다 → 핸들러가 true. 이제 라이브러리가 막아 놨던 그 뒤로가기를 직접 재실행한다.

debug("[Internal] Handler confirmed");
interceptionState.confirmNextNavigation();  // ★ 표식
window.history.go(consumedNavigation.delta);  // go(-1) → 진짜로 A로 이동
왜 nextNavigationConfirmed = true를 먼저 세우나: 곧 부를 go(-1)이 또 popstate를 일으킨다. 이걸 일반 뒤로가기로 오인하면 → 핸들러 또 실행 → 또 승인 → 또 go() → 무한 루프. 이 표식이 "다음 popstate는 사용자 입력이 아니라 내가 방금 승인해서 일으킨 것"이라고 알려, 분기 0에서 핸들러를 건너뛰게 한다.
T6
Browser go(-1) 실행 → 실제로 A로 이동

이번엔 복원이 아니라 진짜 이동. 브라우저가 스택 포인터를 A로 옮긴다.

A {idx:0, τ} ← 진짜로 이동→B {idx:1, τ}
T7
popstate 3차 popstate → 분기 0 (Confirmed) → 통과

go(-1) 완료로 popstate 재발생. 인자는 A의 state {idx:0, τ}.

if (interceptionState.isNextNavigationConfirmed()) {   // ← true! (T5/분기B에서 세움)
  return handleConfirmedNavigation({ nextSessionToken: τ, nextHistoryIndex: 0 });
}

// handleConfirmedNavigation:
interceptionState.consumeConfirmation();  // 표식 리셋
renderedHistoryEntryMetadataStore.set(createRenderedHistoryEntryMetadata({ sessionToken: τ, historyIndex: 0 }));   // renderedMetadata = {idx:0, τ}, sync OFF
return true;   // ★ 이번엔 Next.js가 정상 처리하도록 허용

분기 0이 분기 1·2보다 먼저 평가되므로, 토큰/델타를 따지기 전에 여기서 잡힌다. 핸들러를 재실행하지 않고 true를 반환 → Next 라우터가 정상적으로 A를 렌더한다.

왜 store의 set만으로 충분한가: 우리가 부른 go(-1)으로 이미 존재하던 A 엔트리에 도착했다. A.state에는 P1에서 기록한 올바른 {idx:0, τ}가 그대로 있으므로 history를 다시 쓰지 않고 메모리 포인터만 옮기면 된다.
✓
Done Next.js가 A 렌더 → 뒤로가기 완료
A {idx:0, τ} ← 도착 · 렌더 완료→B {idx:1, τ}
renderedMetadata{ idx: 0, τ } nextNavigationConfirmedfalse (리셋됨) pendingNavigationnull

모든 상태가 A 기준으로 정합. 다음 뒤로가기는 다시 T1부터 깨끗하게 시작된다.

곁가지 — 새로고침-안전 fallback은 언제 쓰이나

T2에서 go(1)과 함께 예약했던 scheduleRestoreCompletionFallback(fallback)(rAF + setTimeout)은 정상 케이스에선 무용지물이다. T4의 delta===0 popstate가 먼저 pendingNavigation를 소비하므로, 뒤늦게 깨어난 fallback은 hasPending() === false를 보고 그냥 끝난다.

fallback이 실제로 일하는 유일한 경우 = 새로고침 직후 첫 액션이 뒤로가기일 때. Next Pages Router는 새로고침 후 우리의 합성 go() 복원에 대해 beforePopState를 건너뛴다(isSsr 가드). 그래서 T4의 follow-up popstate가 아예 안 온다. 이때 fallback이 대신 completeRestoreAndRunHandlers()를 불러 핸들러를 실행한다. consume()의 멱등성 덕에 "둘 중 먼저 도착한 쪽이 이기고 나머진 no-op" — 어느 경로든 핸들러는 정확히 한 번 실행된다.

※ 새로고침이 분기 1(세션 경계)가 아니라 분기 2(Internal)로 처리되는 이유는 P0의 모듈-평가 토큰 복원 때문. 자세히는 popstate-interceptor.ts 분기 2 / history-augmentation-explained.html.
